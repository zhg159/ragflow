# RAGFlow ARM64 构建与启动说明

## 1. 需求概述

### 1.1 背景
- RAGFlow 官方未提供 ARM64 架构的 Docker 镜像
- 需要在 ARM 服务器（如华为云、AWS Graviton 等）上运行 RAGFlow

### 1.2 具体要求
- **架构**: ARM64 (aarch64)
- **嵌入模型**: Qwen/Qwen3-Embedding-0.6B (本地 CPU 模式)
- **部署方式**: Docker Compose
- **镜像仓库**: GitHub Container Registry (ghcr.io)

### 1.3 已知限制
- **Chrome/ChromeDriver 已移除**: 官方仅提供 x86_64 版本，ARM 架构无法使用
  - 影响: 部分依赖浏览器自动化功能可能受限
  - 替代方案: 使用其他文档解析方式

## 3. 构建方法

### 3.1 方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|-----|---------|------|------|
| **GitHub Actions + ARM Runner** | 推荐方案 | 原生 ARM 构建，速度快 | 需要 GitHub 账号和配置 |
| 本地 x86 + QEMU | 临时测试 | 无需额外配置 | 极慢（10倍以上耗时），不推荐 |
| ARM 服务器本地构建 | 有 ARM 机器时 | 速度快，可控 | 需要 ARM 硬件 |

### 3.2 GitHub Actions 构建（推荐）

#### 前置条件
1. Fork RAGFlow 仓库到个人账号
2. 获取 GitHub Personal Access Token (PAT)
3. 配置仓库为私密（可选）

#### 配置步骤

**1. Fork 仓库**
```bash
# 在 GitHub 网页上点击 Fork 按钮
# 或使用 gh CLI
gh repo fork infiniflow/ragflow --fork-name ragflow --clone=false
```

**2. 设置仓库为私密**
```bash
# 进入仓库 Settings -> General -> Danger Zone -> Change visibility
# 选择 "Make private"
```

**3. 配置 GitHub Actions 权限**
- 进入仓库 Settings -> Actions -> General
- 选择 "Read and write permissions"
- 勾选 "Allow GitHub Actions to create and approve pull requests"

**4. 触发构建**
```bash
# 方式1: 推送到 main 分支自动触发
git push origin main

# 方式2: 手动触发
# 进入仓库 Actions -> Build ARM64 Image -> Run workflow
```

#### 构建工作流配置

文件: `.github/workflows/build-arm.yaml`

```yaml
name: Build ARM64 Image

on:
  push:
    branches:
      - main
      - master
  workflow_dispatch:  # 允许手动触发

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build:
    runs-on: ubuntu-24.04-arm  # ARM runner
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install Python dependencies
        run: |
          pip install nltk huggingface-hub

      - name: Download dependencies
        run: |
          python download_deps.py

      - name: List downloaded files
        run: |
          ls -la *.tar.gz *.jar *.deb *.tiktoken 2>/dev/null || true
          ls -la nltk_data huggingface.co 2>/dev/null || true

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build deps image
        run: |
          docker build -f Dockerfile.deps -t infiniflow/ragflow_deps:latest .
          docker tag infiniflow/ragflow_deps:latest ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}-deps:arm64
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}-deps:arm64

      - name: Build main image
        run: |
          docker build -f Dockerfile -t ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:arm64 .
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:arm64

      - name: Export image
        run: |
          docker save ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:arm64 | gzip > ragflow-arm64.tar.gz

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: ragflow-arm64-image
          path: ragflow-arm64.tar.gz
          retention-days: 7
```

#### 构建时间估算

| 阶段 | 预估时间 |
|-----|---------|
| 下载依赖 (download_deps.py) | 5-15 分钟 |
| 构建 deps 镜像 | 1-2 分钟 |
| 构建主镜像 (基础层) | 10-20 分钟 |
| 构建主镜像 (Python 依赖) | 20-40 分钟 |
| 构建前端 (npm build) | 5-10 分钟 |
| **总计** | **40-90 分钟** |

### 3.3 本地构建（ARM 服务器）

如果有 ARM64 服务器，可以直接在本地构建：

```bash
# 1. 克隆仓库

# 5. 构建主镜像

```

## 4. 部署启动

### 4.1 环境配置

文件: `docker/.env`

```bash
# 文档引擎: elasticsearch 或 infinity
DOC_ENGINE=elasticsearch

# 运行设备: cpu 或 cuda
DEVICE=cpu

# 启用 TEI CPU 嵌入服务
COMPOSE_PROFILES=${DOC_ENGINE},${DEVICE},tei-cpu

# 嵌入模型配置
TEI_MODEL=Qwen/Qwen3-Embedding-0.6B
HF_ENDPOINT=https://hf-mirror.com

# 使用自定义 ARM64 镜像
RAGFLOW_IMAGE=ghcr.io/zhg159/ragflow:arm64

# MySQL 配置
MYSQL_PASSWORD=infini_rag_flow

# Elasticsearch 配置
ELASTIC_PASSWORD=infini_rag_flow

# MinIO 配置
MINIO_USER=rag_flow
MINIO_PASSWORD=infini_rag_flow

# Redis 配置
REDIS_PASSWORD=infini_rag_flow
```

### 4.2 启动服务

**方式1: 使用 GitHub 镜像（推荐）**

```bash
# 1. 登录 GitHub Container Registry
echo $GITHUB_TOKEN | docker login ghcr.io -u zhg159 --password-stdin

# 2. 拉取镜像
docker pull ghcr.io/zhg159/ragflow:arm64

# 3. 启动服务
cd docker
docker compose -f docker-compose.yml up -d
```

**方式2: 使用导出的 tar.gz 文件**

```bash
# 1. 从 GitHub Actions 下载 ragflow-arm64.tar.gz
# 进入 Actions -> Build ARM64 Image -> 最新运行 -> Artifacts

# 2. 复制到 ARM 服务器并导入
scp ragflow-arm64.tar.gz user@arm-server:/path/to/
ssh user@arm-server
gunzip -c ragflow-arm64.tar.gz | docker load

# 3. 标记镜像
docker tag ghcr.io/zhg159/ragflow:arm64 infiniflow/ragflow:arm64

# 4. 启动服务
cd /path/to/ragflow/docker
docker compose -f docker-compose.yml up -d
```

## 6. 常见问题

### Q1: 为什么本地 x86 构建 ARM 镜像会卡住？
A: QEMU 模拟 ARM 架构性能极差，前端构建（Vite）需要大量计算，建议使用 GitHub Actions ARM Runner 或原生 ARM 服务器。

### Q2: Chrome/ChromeDriver 移除后有什么影响？
A: 部分依赖 Selenium 的文档解析功能可能受限，但核心 RAG 功能（文档解析、嵌入、检索、生成）不受影响。

### Q3: 如何更新嵌入模型？
A: 修改 `docker/.env` 中的 `TEI_MODEL` 变量，然后重启 tei-cpu 服务：
```bash
docker compose -f docker-compose.yml up -d tei-cpu
```

### Q4: 构建失败如何排查？
A:
1. 检查 GitHub Actions 日志
2. 确认 `download_deps.py` 成功下载所有依赖
3. 验证 Dockerfile 中的镜像引用正确
4. 检查 GitHub Token 权限
